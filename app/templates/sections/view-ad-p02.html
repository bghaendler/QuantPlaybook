Algorithmic Adjoint Differentiation — Part 02

Operator Overloading vs. Source-to-Source Compilers

The two engineering routes to AD compared on speed, memory, intrusiveness and maintainability — and how production quant libraries actually choose.

1. Operator Overloading (OO)

Replace the numeric type: double → adouble (an "active" type). Every arithmetic operator and math function is overloaded to additionally either propagate a tangent (forward mode, no tape) or record the operation onto a tape for later reversal (adjoint mode).

  • Pros: minimal code change (templatize on the scalar type); naturally handles branches, loops, virtual calls — whatever executes gets differentiated; tools: dco/c++, CppAD, ADOL-C, Stan Math; Python autograd/JAX/PyTorch are OO at the tensor level.
  • Cons: runtime overhead (each op does extra work + memory traffic); the tape can be huge (per-operation records — a Monte Carlo XVA run can tape billions of ops: Part 06's memory problem); blocks some compiler optimizations (vectorization through an opaque type).
  • Typical overhead: well-engineered C++ OO adjoints run ~3–10× a double-only valuation; naive implementations 50×+ — implementation quality dominates the asymptotics.

2. Source-Code Transformation (SCT)

A tool (or compiler pass) reads the program and generates new source: tangent-linear code (forward) or adjoint code with explicit reversal (Part 07's structure). Lineage: ADIFOR, Tapenade, OpenAD; modern incarnations: Enzyme (LLVM-level AD), JAX's XLA tracing (a hybrid: trace then compile).

  • Pros: the generated code is plain code — the compiler optimizes it fully (vectorization, inlining); no per-op tape (control flow re-executed or recorded compactly); typically the fastest adjoints.
  • Cons: the tool must parse and analyze your whole language subset — templates, pointers, virtual dispatch and external libraries are hard; build-system integration is invasive; debugging generated code is its own skill.
  • Best fit: numerical kernels with regular structure (PDE steppers, Monte Carlo loops) — exactly where the speed matters.

4. How Production Systems Choose

  • Banks' C++ libraries: predominantly OO (dco/c++-style) for adoption across million-line codebases — with hand-written adjoints for hot kernels and linear-algebra/solver primitives (the "symbolic adjoint of the solve, not of the solver iterations" rule: differentiate $Ax = b$ via $\bar{b} = A^{-\top}\bar{x}$, never through the LU steps).
  • New code / research: JAX-style trace-and-compile gives OO ergonomics with SCT-class performance — increasingly the default for new quant models.
  • Hybrid pattern (the practical answer): OO as the backbone; custom adjoints at module boundaries (curve build, root-finders, regressions); SCT/Enzyme for the few kernels where the last 5× matters.
  • Validation either way: tangent-vs-bump and adjoint-vs-tangent dot-product tests ($\langle \dot{y}, \bar{y}\rangle = \langle \dot{x}, \bar{x}\rangle$) wired into CI — Part 11's discipline.

Key Takeaways

  • OO retypes scalars and records a tape: easy in, memory-hungry, very dynamic-code-friendly.
  • SCT generates derivative source: fastest and leanest, but toolchain-invasive and static-analysis-bound.
  • Differentiate mathematical operations (solves, roots) symbolically at boundaries — never through iteration internals.
  • Production reality is hybrid: OO backbone + custom adjoints + compiled kernels, all under dot-product CI tests.
