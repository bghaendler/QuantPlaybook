Customizing LLMs for Quant Finance Applications

Adapting large language models to finance: the customization ladder from prompting to fine-tuning, retrieval-augmented generation, and where LLMs help versus where they must not be trusted.

1. The Customization Ladder

  • Prompt engineering: few-shot examples, structured output, chain-of-thought — cheapest, surprisingly effective for extraction/classification; the right first try.
  • Retrieval-Augmented Generation (RAG): embed your documents (filings, research, term sheets), retrieve relevant chunks at query time, ground the LLM's answer in them — reduces hallucination, keeps knowledge current without retraining; the dominant enterprise pattern.
  • Fine-tuning: adapt weights on domain data (financial language, house style) — better format adherence and jargon; parameter-efficient methods (LoRA) make it affordable. Needed less often than assumed once RAG is in place.
  • Tool use / agents: the LLM calls pricers, databases, calculators — offloading exactly the quantitative reasoning it's bad at to code that's reliable (the crucial architectural insight).

2. Strong Use Cases

  • Document intelligence: extracting terms from prospectuses/ISDA confirms, summarizing research, parsing earnings calls — the NLP-SDG pipeline supercharged.
  • Research acceleration: code generation for backtests/pricers (validated, never trusted blind), literature synthesis, hypothesis brainstorming.
  • Sentiment & event extraction feeding signals (the sentiment-portfolio talk) — LLMs read context better than bag-of-words sentiment.
  • Interfaces: natural-language query over risk/position systems — "what's my CVA sensitivity to EUR rates?" routed to the real engine via tool use.

3. Where LLMs Must Not Be Trusted

  • Numerical computation: LLMs are unreliable at arithmetic and exact calculation — always route pricing/risk math to deterministic code (tool use), never the model's "mental math."
  • Hallucination: confident fabrication of facts, citations, numbers — unacceptable for anything booked or reported; RAG with source attribution and human review is mandatory.
  • Look-ahead/data leakage: a model trained on data through last year "predicting" returns is leaking — backtests with LLM signals need strict point-in-time discipline (portfolio-workshop's sin).
  • Governance: non-determinism, prompt injection, and explainability gaps make LLMs a model-risk category of their own (the AI-risk talk) — validate, log, and constrain.

Key Takeaways

  • Climb the cheapest rung that works: prompting → RAG → fine-tuning → tool-using agents.
  • RAG grounds answers in your documents; tool use offloads quantitative reasoning to reliable code — the two architectural keys.
  • Best at language tasks (extraction, summarization, code drafting); never trusted for exact computation or unverified facts.
  • LLMs are their own model-risk category: hallucination, leakage, non-determinism — deploy with validation, attribution and human review.
