Algorithmic Adjoint Differentiation

Code Overloading for Backward Interpretation

Building an adjoint by operator overloading: an active type that records a tape during execution, then a backward interpreter that replays it — a minimal working AAD engine.

1. Architecture of an OO Adjoint

Three components, ~200 lines for a usable core:

  • Active type ADouble: holds a value + a node index into the tape.
  • Tape: append-only array of records — op code (or just local partials) + parent indices. Recording is the forward execution: overloaded *, +, exp, log, ... compute the value and append a node.
  • Backward interpreter: allocate an adjoint per node, seed the output's adjoint with 1, walk the tape backwards applying $\bar{v}_{parent} \mathrel{+}= w \cdot \bar{v}_{node}$ for each stored partial $w$.

Storing the partials (not op codes) at record time is "preaccumulation at the elemental level" — the interpreter then needs no primal values at all, trading slightly more tape per node for a simpler, faster reverse sweep.

2. Minimal Python Implementation

tape = []                                   # records: (parents, partials)

class ADouble:
    __slots__ = ('val', 'idx')
    def __init__(self, val, idx=None):
        self.val = val
        self.idx = idx if idx is not None else new_node((), ())
    def __mul__(self, o):
        o = wrap(o)
        return ADouble(self.val*o.val, new_node((self.idx, o.idx), (o.val, self.val)))
    def __add__(self, o):
        o = wrap(o)
        return ADouble(self.val+o.val, new_node((self.idx, o.idx), (1.0, 1.0)))

def exp(x): v = math.exp(x.val); return ADouble(v, new_node((x.idx,), (v,)))
def new_node(parents, partials): tape.append((parents, partials)); return len(tape)-1

def grad(out):                              # backward interpreter
    adj = [0.0]*len(tape); adj[out.idx] = 1.0
    for i in range(out.idx, -1, -1):
        parents, partials = tape[i]
        for p, w in zip(parents, partials):
            adj[p] += w * adj[i]            # the += of Part 03
    return adj                              # read inputs' adjoints by their idx

Usage: build inputs as ADouble, run the unchanged pricer source (duck typing), call grad(PV) — every input's sensitivity drops out. This 30-line engine is, structurally, PyTorch's autograd.

3. Production Concerns the Toy Hides

  • Tape memory layout: contiguous flat arrays (struct-of-arrays) for parents/partials — cache-friendly reverse sweeps; per-object Python lists are 100× off the pace (use C++/NumPy blocks; CppAD/dco do exactly this).
  • Tape lifecycle: per-path tape reset (Part 06's pathwise trick) — tape.clear() between paths keeps memory flat; thread-local tapes for parallelism.
  • Passive fast path: a template/dual-build so the same source runs pure-double when no derivatives are wanted — never pay tape cost in plain pricing.
  • Custom nodes: register super-node ops (solve, interpolate, BS formula) that append one record with analytic partials (Parts 02–03) — the single biggest speed/memory lever.
  • Branches recorded implicitly: the tape stores what executed — re-running with different inputs requires re-taping (contrast JAX's traced-and-compiled static graphs; dynamic taping is the price of Python-style flexibility).

4. Checkpointed Interpretation

For long simulations, combine the interpreter with Part 06's checkpoints: tape only the current segment; at segment boundaries, store state snapshots; reverse segment-by-segment, re-taping each from its snapshot. The interpreter doesn't change — only the driver loop does. This composition (OO recording + Revolve scheduling + custom nodes) is, component for component, the architecture of commercial AAD libraries deployed for XVA.

Key Takeaways

  • OO adjoint = active type + append-only tape + backward interpreter; storing local partials simplifies the reverse sweep.
  • A 30-line engine demonstrates the whole mechanism — and is structurally identical to autograd.
  • Production = flat memory layout, per-path tape resets, passive fast path, custom super-nodes.
  • Checkpointing changes the driver, not the interpreter — the components compose into the commercial AAD stack.
