Algorithmic Adjoint Differentiation — Part 04

Implementing Manual Tangent-Linear Code

Writing forward-mode AD by hand: the line-by-line discipline, control flow rules, and a full tangent-linear Monte Carlo pricer.

1. The Mechanical Rules

For every active variable v, introduce v_d (its tangent, $\dot{v}$). For every assignment, write the differentiated line first (it reads old values), then the original:

// original              // tangent-linear (before the original line)
v = a * b;               v_d = a_d * b + a * b_d;
w = exp(v);              w_d = exp(v) * v_d;        // or: w_d = w * v_d AFTER w computed
u = v / w;               u_d = (v_d - u * w_d) / w; // reuse u: cheaper & stabler
s = sqrt(v);             s_d = 0.5 / s * v_d;       // guard v=0 like the primal must
  • Seeding: to get $\partial y/\partial x_k$, set xk_d = 1, all other input tangents 0; outputs' tangents are the column of the Jacobian.
  • Re-use computed primals (the u, s tricks): halves the flops and mirrors what good AD tools emit.
  • Passive variables (ints, flags, anything provably derivative-free) get no tangent — "activity analysis" by hand.

2. Control Flow

  • Branches: differentiate each branch's body; the condition itself is not differentiated (it's piecewise definition). Mind the kinks: max(x,0) has tangent x > 0 ? x_d : 0 — correct a.e., undefined at the kink (the payoff-smoothing issue Part 09 revisits).
  • Loops: the tangent loop runs in step with the primal loop, same iteration count — forward mode never reverses anything (the adjoint's reversal pain, Part 07, doesn't exist here).
  • Function calls: each function gets a tangent version f_d(x, x_d, &y, &y_d) computing value and tangent — composability by signature convention (Tapenade's _d convention).
  • Iterative solvers: don't differentiate the iterations; differentiate the solved equation: from $F(x, \theta) = 0$, $\;\dot{x} = -F_x^{-1}F_\theta\dot{\theta}$ at the converged point (implicit function theorem) — exact, cheap, and avoids the convergence-of-derivatives question.

3. Worked: Tangent-Linear Monte Carlo (Vega of a Call)

def mc_call_vega(S0, K, T, r, sigma, Z):           # Z: fixed normals (CRN!)
    drift   = (r - 0.5*sigma**2)*T
    drift_d = -sigma*T                              # d(drift)/d(sigma)
    vol     = sigma*np.sqrt(T)
    vol_d   = np.sqrt(T)
    ST     = S0*np.exp(drift + vol*Z)
    ST_d   = ST*(drift_d + vol_d*Z)                 # chain rule through exp
    pay    = np.maximum(ST - K, 0.0)
    pay_d  = np.where(ST > K, ST_d, 0.0)            # kink rule
    disc   = np.exp(-r*T)
    return disc*pay.mean(), disc*pay_d.mean()       # price, vega

Checks that should pass: vega vs. central bump ($\sigma \pm 10^{-5}$) to ~10 digits using the same Z (common random numbers — pathwise derivatives require the randomness held fixed); and vs. BS analytic vega within MC error. This is the pathwise Greek method of the Monte Carlo literature — tangent-linear AD and pathwise sensitivities are the same thing.

4. When Manual Forward Mode Is the Right Tool

  • Few inputs, many outputs: curve bumps through a cash-flow generator, scenario engines, $\partial(\text{whole smile})/\partial(\text{one model param})$ in calibration Jacobians.
  • As the validation layer for adjoints: the dot-product identity $\langle\dot y, \bar y\rangle = \langle\dot x, \bar x\rangle$ tests both implementations against each other (Part 11).
  • Small kernels where introducing a tool dependency isn't worth it — the rules above are entirely mechanical and code-reviewable.
  • Complex-step shortcut for a single derivative of real-analytic code: $f'(x) \approx \mathrm{Im}\,f(x + ih)/h$, $h = 10^{-20}$ — forward mode smuggled through the complex type, no cancellation (CA Part 11's trick).

Key Takeaways

  • Tangent line before primal line, reuse primal results, seed one input at a time.
  • Branches differentiate piecewise (mind kinks); loops stay in step; solvers differentiate the equation, not the iterations.
  • Tangent-linear MC with frozen randomness = pathwise Greeks; validate against bumps with CRN.
  • Forward mode is the right tool for few-input maps and as the adjoint's CI cross-check.
