GPU Acceleration of Monte Carlo (CVA & CCR)

Porting counterparty-risk simulation to the GPU: the SIMT execution model, what maps well, and the memory/RNG/reduction patterns that turn overnight batches into minutes.

1. Why CVA/CCR Is a GPU Problem

Counterparty credit risk needs the exposure profile of a whole netting set across time:

$$\mathrm{CVA} = (1 - R)\int_0^T \mathrm{EE}(t)\,dPD(t), \qquad \mathrm{EE}(t) = \mathbb{E}\big[(\textstyle\sum_i V_i(t))^+\big],$$

computed by simulating thousands of market paths, revaluing the entire portfolio at dozens of future dates per path. The work is embarrassingly parallel across paths (RSI/ARM independence) and arithmetically dense — exactly the SIMT sweet spot. Banks run these as overnight batches on CPU farms; GPU porting compresses them to intraday, enabling live CVA risk (the AAD masterclass's gradients on top).

2. The SIMT Execution Model

  • Threads → paths: one thread (or warp) per Monte Carlo path; thousands run concurrently. The natural decomposition needs almost no rethinking — the path loop becomes the grid.
  • Warps of 32 lockstep threads: all execute the same instruction — branch divergence (paths taking different if branches) serializes and kills throughput. Restructure payoff logic to be branch-light (arithmetic masks instead of conditionals).
  • Occupancy: enough resident warps to hide memory latency — bounded by registers/shared-memory per thread; pricing kernels with large state need careful budgeting.
  • Memory hierarchy: coalesced global-memory access (adjacent threads → adjacent addresses) is mandatory; market data in constant/shared memory; the per-path state in registers.

3. The Three Porting Patterns That Matter

  • Counter-based RNG (the enabling trick): Philox/Threefry (ARM Part 09) generate $\text{rand}(\text{path}, \text{step})$ statelessly from indices — no per-thread RNG state, perfect reproducibility, and independent substreams for free. This is why GPU MC is trustworthy: bit-identical to CPU given the same indices.
  • Layout = correlated draws + revaluation: store paths as (steps × paths) column-major so step-vectors are coalesced; Cholesky-correlate (ARM Part 08) and apply pricing kernels with adjacent threads on adjacent paths.
  • Parallel reduction for EE(t): averaging $(\sum_i V_i)^+$ across paths is a tree reduction in shared memory — and must be deterministic (fixed pairwise order) if governance requires bit-reproducibility (ARM/AAD theme); naive atomicAdd is fast but non-reproducible.

4. Practical Workshop Notes

  • Profile before porting: revaluation usually dominates — port the pricing inner loop first; leave I/O and aggregation on the host initially.
  • Precision: FP32 is ~2× FP64 throughput on many GPUs but CVA sums can lose accuracy — use FP64 for accumulation, FP32 for path generation where validated, or Kahan summation.
  • Validate relentlessly: GPU result must match the CPU reference within Monte Carlo error (same counter-RNG indices ⇒ near bit-identical); divergence flags a race or reduction-order bug.
  • AAD on GPU: pathwise adjoints (AD masterclass) port directly — per-path tapes fit in registers/shared memory (AD Part 06's budget), giving CVA and its full sensitivity vector in one pass; Enzyme/Philox compose cleanly.
  • Realistic speedups: 20–100× vs. single CPU core on the revaluation-heavy kernels; the win is wall-clock (overnight → minutes), enabling intraday/real-time CCR.

Key Takeaways

  • CVA/CCR is path-parallel and revaluation-dense — the canonical GPU Monte Carlo workload.
  • SIMT: threads = paths; avoid branch divergence, ensure coalesced memory, manage occupancy.
  • Counter-based RNGs (Philox) give stateless, reproducible, independent streams — the key to trustworthy GPU MC.
  • Deterministic reductions, FP64 accumulation, and pathwise AAD-on-GPU deliver CVA + Greeks at 20–100× with validated accuracy.
