Algorithmic Adjoint Differentiation

Tangent-Linear Efficiency: Vector Mode & Sparsity

Making forward mode pay: propagate many directions at once, exploit Jacobian sparsity via graph coloring, and amortize the primal work.

1. Vector (Multi-Directional) Mode

Naive forward mode re-runs the primal once per input. Vector mode carries a block of tangents $\dot{V} \in \mathbb{R}^{1 \times p}$ per variable — one primal execution, $p$ directions:

w = exp(v)        →   w_d[0..p-1] = w * v_d[0..p-1]     // one primal exp, p mults
  • Why it's faster than p sweeps: the primal values ($\exp$, $\ln$, divisions — the expensive parts) are computed once; the per-direction work is mostly fused multiply-adds over contiguous arrays — SIMD-perfect. Speedups of 3–10× over repeated single sweeps are routine at $p$ = 8–32.
  • Jacobian assembly: seed $\dot{X} = I_n$ (in blocks of $p$) → full $J$ in $\lceil n/p\rceil$ vectorized passes.
  • This is exactly JAX's jacfwd/vmap pattern and Tapenade's "vector mode" — also how bump grids should have been done all along.

2. Sparsity: Don't Compute Zeros

Real quant Jacobians are sparse: a 40-bucket curve build where each swap touches ~4 buckets; a local-vol PDE where each output node depends on a few surface nodes. Compression by coloring:

  • Two inputs whose columns share no nonzero row can be seeded together (their effects never overlap) — finding the minimal seed count is graph coloring on the column-intersection graph (Curtis-Powell-Reid).
  • Result: a banded/structured $n$-column Jacobian costs χ (chromatic number) sweeps, often 3–10 regardless of $n$ — tridiagonal needs 3 colors, period.
  • Workflow: detect the sparsity pattern (one boolean propagation pass or static analysis) → color → seed compressed blocks → scatter back into $J$.
  • The same compression applies to bumping (it predates AD — "compressed finite differences") and to reverse mode by rows.

3. Quant-Library Plays

  • Calibration Jacobians: Levenberg-Marquardt over (instruments × parameters) — parameters are few: vector-forward over all of them in one pass beats adjoint (which would sweep once per instrument residual). Mode choice by shape, as promised in Part 01.
  • Curve risk through cash-flow generators: bucket sparsity (each trade hits few buckets) + coloring ⇒ full bucketed deltas at ~constant cost in bucket count.
  • Scenario/"what-if" engines: the $p$ directions need not be coordinate bumps — seed with scenario vectors (parallel shift, twist, butterfly): vector-forward returns the portfolio's directional derivatives for the whole scenario set in one valuation.
  • Stochastic-process tangents: per-path tangents over $p$ parameters vectorize across both paths and directions — a (paths × p) tangent matrix in NumPy/CUDA, the layout the GPU workshop assumes.

4. Cost Model & Crossover

Rule-of-thumb flop counts per primal cost 1: single tangent ≈ 2–2.5; vector-$p$ tangent ≈ 1 + 0.15p (amortized); adjoint ≈ 3–5 (+ memory traffic). Crossover heuristics:

  • $n \lesssim 20$ inputs (or χ colors ≲ 20): vector forward wins on simplicity and speed.
  • $n$ in the hundreds+ with one PV: adjoint, no contest (Part 06 onward).
  • Both large (full dense Jacobian): you probably don't need the full Jacobian — re-ask the question (directional scenarios? sparsity? low-rank structure?).

Key Takeaways

  • Vector mode amortizes the primal: one execution, $p$ SIMD-friendly tangent directions.
  • Sparsity + coloring compresses $n$ columns into χ seeds — bucketed curve risk at constant cost.
  • Seed with scenario vectors, not just coordinates — directional risk is forward mode's native product.
  • Crossover: few inputs → vector forward; many inputs, one output → adjoint; both huge → restructure the question.
