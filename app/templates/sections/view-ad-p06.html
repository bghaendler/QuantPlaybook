Algorithmic Adjoint Differentiation

Adjoint Code Mechanics & Memory Constraints

The price of the O(1) gradient: reverse-order data dependencies, tape growth, and checkpointing — the engineering that makes adjoints feasible at Monte Carlo scale.

1. Why Reverse Needs Memory

The adjoint of w = exp(v) is v̄ += w·w̄ — it needs w (a primal value) at a moment when execution runs backwards. Every nonlinear operation's adjoint references primal state, so the forward sweep must preserve it:

  • Tape (store-all): record needed values (and, in OO tools, the op sequence) during the forward pass; replay backwards. Memory grows linearly with operations executed: a 100k-path, 252-step MC with ~50 ops/step ≈ 10⁹+ records — tens of GB. Memory bandwidth, not flops, becomes the adjoint's bottleneck.
  • Recompute-all: store nothing; recompute forward from the start for every needed value — $O(T^2)$ time. Both extremes lose; the answer is between them.

2. Checkpointing: Trading Time for Memory

Store full state snapshots at $c$ checkpoints; during reversal, recompute each segment forward from its checkpoint, tape only the active segment:

  • Uniform checkpointing: √-rule — $c \approx \sqrt{T}$ checkpoints give memory $O(\sqrt{T})$ at ~2× forward recomputation.
  • Binomial/Revolve (Griewank-Walther): provably optimal schedule — memory $O(\log T)$ at $O(\log T)$ recompute factor; the standard for long time-stepping (PDE solvers, long-horizon XVA simulations).
  • Online variants handle unknown-length loops (adaptive time-stepping).
  • ML knows this as gradient/activation checkpointing — identical algorithm, different vocabulary.

3. Structural Tricks That Shrink the Tape

  • Pathwise independence (the Monte Carlo gift): paths don't interact until the final average — reverse each path separately and accumulate $\bar\theta$: tape size drops from (paths × steps) to (one path × steps). This single observation makes AAD-XVA practical.
  • Linear segments are free: adjoints of linear ops need no primal values — discounting, summations, matrix products with constant matrices tape nothing but structure.
  • Super-nodes with analytic adjoints (Part 03): a root-solve, a regression ($\hat\beta = (X^\top X)^{-1}X^\top y$ has a clean adjoint), an interpolation — replace thousands of taped elementals with one closed-form reversal.
  • Preaccumulation: collapse straight-line blocks into local Jacobians at record time — fewer, fatter tape entries; standard in dco/c++-class tools.
  • Invertible steps: when the step can be run backwards exactly (some SDE schemes, reversible integrators), recompute primal state in reverse — zero storage for those segments.

4. A Capacity Budget (Order-of-Magnitude)

Adjoint MC pricer, 100k paths × 252 steps, per-path reversal: active doubles per step ≈ 20 → per-path tape ≈ 20 × 252 × 8B ≈ 40 KB — trivially cached; total cost ≈ forward 1× + reverse ~2–3× ⇒ full smile/curve/parameter gradient at ~4× one pricing, in <100 MB. The same job taped monolithically: ~400 GB — infeasible. The lesson generalizes: adjoint feasibility is an architecture property, not a tool property — structure (path independence, super-nodes, checkpoints) is where the engineering lives.

Key Takeaways

  • Nonlinear adjoints consume primal values backwards — store (tape) or recompute (checkpoint).
  • Revolve-style checkpointing: $O(\log T)$ memory for $O(\log T)$ recompute — optimal for time-stepped codes.
  • Per-path reversal, linear-op freebies, analytic super-nodes and preaccumulation shrink tapes by orders of magnitude.
  • Budget memory before coding: adjoint feasibility is decided by program structure, not by the AD library.
