Python Lab Workshop (Code Submissions)

The applied programming track: the scientific Python stack for quants, the lab assignments that recur, and the engineering standards a code submission must meet.

1. The Quant Python Stack

  • NumPy — vectorized arrays; the non-negotiable foundation (no Python loops over paths/strikes). Broadcasting, einsum, and views over copies are the core skills.
  • SciPy — optimize (calibration: least_squares, brentq, minimize), stats (distributions, fitting), integrate, interpolate, linalg (Cholesky, eig).
  • pandas — time series, the merge_asof alignment of HFT Part 11, resampling, rolling windows.
  • matplotlib — smiles, surfaces, P&L distributions, convergence diagnostics.
  • Specialized: QuantLib-Python (curves, instruments — the dev-quantlib-export tool), numba/Cython for hot loops, JAX/PyTorch for AD (the AAD masterclass).

Run any of this live in the Live Python Sandbox section (Pyodide) — it has NumPy loaded.

2. The Recurring Lab Assignments

  1. Black-Scholes + Greeks, then implied vol by Newton/Brent (the inverse problem of MM Part 02) — vectorized over a strike grid.
  2. Monte Carlo pricer with antithetics + control variates (MM Part 09), reporting the standard error — extended to barriers (Brownian-bridge correction) and Asians.
  3. Binomial/PDE pricer for American options (Crank-Nicolson with the free-boundary projection).
  4. Curve bootstrap and risk (the term-structure course) — solve for zero rates, compute bucketed DV01.
  5. Calibration: fit SABR/Heston to a smile via least_squares (ASV masterclass) — Jacobian by AD or vector-bump (AD Part 05).
  6. VaR engine: historical + Monte Carlo (ARM Parts 05, 08), with backtesting (Kupiec).

3. Submission Standards (What Graders Check)

  • Correctness with a benchmark: every numerical result validated against a closed form or independent method (MC vs. analytic, PDE vs. tree) within stated tolerance — never an unchecked number.
  • Vectorization: no per-element Python loops where NumPy applies; a loop over 10⁶ paths is an automatic fail.
  • Reproducibility: seeded RNG (default_rng(seed)), pinned dependencies, deterministic output — the ARM Part 09 / governance habit.
  • Reported error bars: Monte Carlo results quote the standard error; fits report residuals — a point estimate without uncertainty is incomplete.
  • Readability: functions with docstrings stating units and conventions (carry, day-count), type hints, no magic numbers — code reviewed like production.

4. Common Failure Modes (Pre-Submission Checklist)

  • The $-\tfrac12\sigma^2$ drift omission in GBM (MM Part 09) — silently biases everything log-related.
  • Float comparison & kinks: ST > K exactly at the strike; payoff discontinuities breaking pathwise Greeks (AD Part 07).
  • Look-ahead / in-sample leakage in backtests — the cardinal research sin.
  • Non-PSD covariance fed to Cholesky (ARM Part 10) — clip eigenvalues or use Higham first; the correlation-matrix tool shows the failure.
  • Silent broadcasting bugs: a $(n,)$ vs. $(n,1)$ shape mismatch producing an $(n,n)$ accidental outer product — assert shapes.
  • Unseeded randomness making results irreproducible between runs.

Key Takeaways

  • Master NumPy/SciPy/pandas vectorization — it's the foundation every lab is graded against.
  • The assignment arc spans the whole curriculum: BS/Greeks, MC, PDE, curves, calibration, VaR.
  • Submissions need benchmark validation, vectorization, seeded reproducibility, and reported error bars.
  • Avoid the classic traps: missing drift term, kink/float comparisons, look-ahead, non-PSD matrices, unseeded RNG.
