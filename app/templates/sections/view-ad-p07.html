Algorithmic Adjoint Differentiation

Adjoint Reverse-Mode Implementation & Stacks

Writing adjoint code by hand: the statement-reversal rules, the push/pop stack discipline for overwritten variables and control flow, and a complete worked kernel.

1. The Reversal Rules

For each primal assignment, the adjoint statements run in reverse program order; each v = φ(a, b) becomes:

ā += (∂φ/∂a)·v̄ ;   b̄ += (∂φ/∂b)·v̄ ;   v̄ = 0   // v consumed: reset its adjoint
  • Always += (fan-out accumulates — Part 03's rule); reset the assigned variable's adjoint after distributing it (it was overwritten in the primal, so its old adjoint chain ends here). Missed resets corrupt gradients in loops — the second-most-common bug after missed +=.
  • In-place updates: v = v*a ⇒ ā += v_old·v̄; v̄ = a·v̄ — needs the old v: store it (stack) before overwriting.

2. Stacks: The Reversal Data Structure

Reversal is LIFO by nature, so the canonical implementation pairs every forward PUSH with a backward POP:

  • Value stack: push any value about to be overwritten that the adjoint will need (Tapenade's PUSHREAL/POPREAL); pop restores it just before the corresponding adjoint statement.
  • Control-flow stack: push which branch was taken / how many loop iterations ran; the backward sweep pops to replay decisions in reverse — if reverses to the same branch's adjoint, a for of $n$ iterations reverses as $n$ adjoint iterations in reverse order.
  • Call stack discipline: each function gets f_fwd (primal + pushes) and f_bwd (pops + adjoints); composite reversal nests like brackets. This signature pair is exactly JAX's custom_vjp contract.

3. Worked Kernel: Adjoint GBM Loop

# forward (per path): tape S_t before overwrite
S = S0
for t in range(n):
    push(S)                                   # value stack
    S = S * exp(drift + vol*Z[t])
payoff = max(S - K, 0); PV = df * payoff

# reverse
S_bar = df * (1.0 if S > K else 0.0) * PV_bar          # payoff & discount adjoints
drift_bar = vol_bar = 0
for t in reversed(range(n)):
    S_old = pop()
    g = exp(drift + vol*Z[t])                  # recompute (cheap) or push it too
    drift_bar += S_old * g * S_bar             # ∂S/∂drift = S_old·g
    vol_bar   += S_old * g * Z[t] * S_bar
    S_bar      = g * S_bar                     # ∂S/∂S_old
S0_bar = S_bar
# then: sigma_bar = vol_bar*sqrt(T/n) + drift_bar*(-sigma*T/n), r_bar = ...

Validation: $\;S0\_bar$ must equal the pathwise delta (= tangent-linear run, Part 04) and the dot-product test $\langle\dot y,\bar y\rangle = \langle\dot x,\bar x\rangle$ must hold to machine precision for random seeds — the two tests that catch essentially all reversal bugs.

4. Hand-Coding in Practice

  • When to hand-code: hot kernels (inner MC loops, PDE steppers), library boundaries (solvers, regressions — Part 02's symbolic-adjoint rule), and anywhere a tool's tape would blow Part 06's memory budget.
  • Style that prevents bugs: single-assignment locals where possible (no pushes needed); explicit _bar naming; adjoint code blocks mirroring primal blocks in reverse order with comments pairing them.
  • Discontinuities: the digital part of payoffs has zero pathwise derivative almost surely but the expectation's derivative is not zero where the payoff jumps — smooth the payoff or add likelihood-ratio/vibrato terms for digitals and barriers; silent omission biases Greeks (flagged again in Part 09).
  • Accumulation order: floating-point adjoint sums across paths/threads need deterministic reduction (pairwise/Kahan, fixed ordering) if bit-reproducibility is a governance requirement (ARM Part 09's theme).

Key Takeaways

  • Adjoint of each assignment: distribute with +=, then reset the overwritten variable's adjoint.
  • PUSH/POP stacks carry overwritten values and control-flow decisions; loops reverse iteration order.
  • The GBM kernel pattern (tape S, reverse the recursion) is the template for every simulation adjoint.
  • Dot-product + tangent-vs-adjoint tests are non-negotiable; handle payoff discontinuities explicitly.

Plain English Notes

The Big Picture: Implementing Adjoint mode by hand requires reversing the control flow of your code, pushing values onto memory stacks, and accumulating derivatives backward.

  • Always Accumulate: In reverse mode, derivatives are always added ($+=$) to handle variables that are used multiple times.
  • The Stack: Intermediate variables must be stored in a stack during the forward run and popped off in reverse order.
  • Validation: Hand-coded adjoints must be validated using dot-product tests against forward mode to catch accumulation bugs.
